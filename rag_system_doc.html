系统由 7 个功能模块组成，各模块职责单一、接口清晰
📄 知识库构建
PDF 文档解析模块
pdf_parse.py · DataProcess
将汽车用户手册 PDF 解析为文本块，综合使用三种策略确保内容完整性：
  • ParseBlock：按标题分块，保留语义完整性（512/1024 token）
  • ParseAllPage：滑动窗口法，保留跨页上下文（256/512 token）
  • ParseOnePageWithRule：简单切分法（256/512 token）
  • 三种方法 × 两种长度 = 6 轮解析后去重
🔍 向量检索
语义召回模块
retriever/m3e_retriever.py · bge_retriever.py
基于 HuggingFace 嵌入模型 + FAISS 向量库实现稠密向量检索：
  • M3E-Large：中文语义嵌入，归一化余弦相似度
  • BGE-M3：支持跨语言、稀疏+稠密混合嵌入
  • 向量库持久化存储，支持热加载
  • 返回 (Document, score) 对，score 用于无答案判断
🔤 字面召回
稀疏召回模块
retriever/bm25_retriever.py · tfidf_retriever.py
基于词频统计的字面相关性检索，弥补语义召回的关键词漏检问题：
  • BM25：LangChain BM25Retriever，适合长文档词频分布
  • TF-IDF：LangChain TFIDFRetriever，适合关键词精确匹配
  • 与语义召回互补，覆盖不同类型查询
🎯 精排优化
Reranker 重排序模块
rerank_model.py · reRankLLM
对多路召回结果进行二阶精排，提升上下文相关性：
  • BCE-Reranker-Base：百度 BCE 双向交叉编码器
  • BGE-Reranker-Large：BAAI 高精度重排模型
  • 对 (query, doc) 逐对打分 → 降序排列 → 取 Top-K
  • 推理时使用 FP16 + CUDA 加速，用后释放显存
⚡ 推理加速
vLLM 推理模块
vllm_model.py · ChatLLM
基于 vLLM 框架的大模型高性能推理封装：
  • 支持 Qwen2-7B / ChatGLM3-6B / Baichuan2-7B 三种基座
  • PagedAttention + Continuous Batching 优化
  • Beam Search 解码，best_of=2 保障答案质量
  • Batch 推理：5 个 Prompt 一次性并行推理
  • 相比普通推理速度提升约 1×
✨ 提示优化
Prompt 增强 & 构建
generate_answer.py
通过 Query 扩展和 Prompt 工程提升检索与生成质量：
  • Query 扩展：LLM 先对问题生成摘要答案，拼接原始 query 增强语义覆盖
  • Prompt 模板：约束模型基于已知内容回答，禁止编造，输出中文
  • 动态 top_k 与 max_length 控制 Context 长度
  • 距离阈值 > 500 时输出"无答案"
📈 效果评估
评估指标模块
test_score.py
双维度自动化评估体系：
  • 语义相似度：text2vec 向量化后 cosine 相似度（50%权重）
  • 关键词覆盖：Jaccard 系数匹配标准答案关键词（50%权重）
  • 综合得分 = 0.5×语义 + 0.5×关键词覆盖率
  • 评估结果持久化至 metrics.json

Q1 请介绍一下 RAG 的基本原理，以及为什么要用 RAG 而不是直接用 LLM 回答问题？基础
RAG（Retrieval-Augmented Generation） 即检索增强生成，核心思路是：先从外部知识库中检索出与用户问题最相关的文档片段，再将这些片段作为上下文（Context）连同问题一起输入 LLM，引导模型基于这些内容生成答案。

为什么不直接用 LLM？
  • 知识时效性：LLM 训练数据有截止日期，无法覆盖最新或私有领域知识（如特定车型手册）；
  • 幻觉问题：LLM 在没有上下文支撑时容易编造事实，RAG 通过提供文档依据可显著减少幻觉；
  • 可溯源性：RAG 的答案可以对应到具体文档块，便于核实和溯源；
  • 成本：微调私有 LLM 成本极高，RAG 只需维护知识库即可低成本更新知识。
Q2 你的项目中为什么要使用多路召回，而不只用一种检索方式？中等
这是检索系统中的经典"召回融合"策略，原因有两点：

  • 语义召回（M3E / BGE）：捕捉语义层面的相关性，泛化能力强，适合"靠背太热怎么办"→"座椅加热"这类表述不同但语义相近的查询；但对专有名词、精确关键词的命中率较低。
  • 字面召回（BM25 / TF-IDF）：基于词频统计，对关键词精确命中能力强，适合"危险警告灯"这类精确名词检索；但无法处理同义词和表述变体。
两类方法互补，混合使用后总召回覆盖率显著高于单路。在此项目中，4 路 × Top-15 = 60 个候选，再经 Reranker 精排取 Top-6，实现"宽召回 + 精精排"的两阶段检索范式。
Q3 什么是 Reranker？它和 Embedding 召回的区别是什么？为什么引入 Reranker 能提升效果？中等
  • Embedding 召回（Bi-Encoder）：将 query 和 document 分别编码为向量，计算向量距离进行检索。优点是速度快（预计算 doc 向量），缺点是 query 和 doc 没有深度交互，语义理解有限。
  • Reranker（Cross-Encoder）：将 (query, doc) 拼接后一起送入模型，进行深度交叉注意力计算，逐对打分。优点是精度高，缺点是无法预计算、推理慢，因此只用于候选集精排而非全量检索。
引入 Reranker 的核心价值在于：缩短 LLM 的上下文窗口中无关内容，让模型更聚焦于真正相关的文档，同时节省 token 推理成本，降低"lost in the middle"问题（LLM 对长 Context 中间位置注意力下降）。
Q4 你的 PDF 解析用了三种方法，为什么不能只用一种？各有什么优缺点？中等
  • ParseBlock（分块解析）：利用 pdfplumber 识别字体大小变化来区分标题和正文，尽量把"小标题+内容"放在同一文档块。优点：语义完整；缺点：依赖 PDF 排版规律，部分扫描件或排版复杂的 PDF 效果不稳定。
  • ParseAllPage + SlidingWindow（滑窗）：按句号切分整个文档后用滑动窗口连接，天然解决跨页断句问题。优点：跨页连续性好；缺点：可能引入跨章节的噪声拼接。
  • ParseOnePageWithRule（非滑窗）：逐页提取文本，超过阈值则按句号均匀切分。优点：简单稳定；缺点：跨页内容会被割断。
三种方法从不同角度分割文档，互为补充。经过去重后，能最大化覆盖所有潜在相关文本块，提升召回率。
Q5 什么是 vLLM？它如何提升推理速度？PagedAttention 是什么原理？进阶
vLLM 是一个专为 LLM 推理设计的高性能服务框架，核心技术包括：

  • PagedAttention：受操作系统虚拟内存分页管理启发。传统 KV Cache 需要预分配连续 GPU 显存，碎片严重；PagedAttention 将 KV Cache 分成非连续的"页"动态分配，显存利用率大幅提升（减少约 55% 浪费），允许更大 Batch Size。
  • Continuous Batching（连续批处理）：传统静态批处理需等最长序列完成，vLLM 实现动态插入新请求，GPU 利用率从不足 40% 提升到接近 100%。
  • CUDA Kernel 优化：使用高效的注意力计算核，减少内存带宽瓶颈。
在本项目中，还利用了 vLLM 的 批量推理（Batch Inference）能力，5 个 Prompt 一次推理，进一步降低整体延迟。
Q6 什么是 Prompt 增强（Query Expansion）？你是怎么做的，有什么依据？中等
这里使用的是类似 HyDE（Hypothetical Document Embeddings）的变体思路：

  • 原始 Query 可能表述简短（如"靠背热怎么办"），向量化后在语义空间中与标准文档片段距离较远；
  • 先让 LLM 对问题生成一个简洁的假设性答案（不超过16字），如"可以关闭座椅加热功能"；
  • 将假设答案拼接到原始 Query，形成检索词"靠背热怎么办 可以关闭座椅加热功能"，其嵌入向量更接近真实文档空间；
  • 以此作为实际的检索 Query，显著提升语义召回的相关性。
代价是多一次 LLM 调用，但由于该推理 prompt 极短，开销可接受。
Q7 FAISS 是什么？为什么选择 FAISS 而不是其他向量数据库？基础
FAISS（Facebook AI Similarity Search） 是 Meta 开源的向量相似度检索库，专门针对高维向量的近似最近邻（ANN）搜索进行优化。

选择 FAISS 的原因：
  • 轻量无依赖：本地运行，无需单独部署数据库服务，适合实验性项目；
  • 性能高：支持 GPU 加速、多种索引类型（Flat / IVF / HNSW）；
  • LangChain 原生支持：与 HuggingFace Embeddings 无缝集成；
  • 持久化：支持 save_local / load_local，避免每次重新构建索引。
与 Milvus / Qdrant / Pinecone 等相比，FAISS 更适合单机原型验证场景；生产环境中可替换为 Milvus 等支持分布式的方案。
Q8 你的评估指标是语义相似度 + 关键词覆盖率各 50%，为什么这样设计？有没有局限性？中等
设计原因：
  • 语义相似度（text2vec cosine）：衡量答案整体语义是否接近标准答案，能识别同义改写；但对专有名词、数字等关键信息不敏感。
  • 关键词 Jaccard 覆盖率：确保预测答案包含标准答案中的核心关键词（如"座椅加热""危险警告灯"），弥补纯语义评估的短板。
  • 两者 5:5 组合是一种相对均衡的方式，既考量语义流畅性，又保障关键信息覆盖。
局限性：
  • 该评估依赖预标注的关键词，人工成本较高；
  • 不能评估答案的逻辑连贯性和安全性；
  • 更完善的方案可引入 RAGAS、G-Eval 等 LLM-based 自动评估框架，或加入 BLEU/ROUGE 等。
Q9 如果让你优化这个系统，你会从哪些方向入手？进阶
  • 知识库质量优化：引入表格、图片 OCR 解析（当前仅文本）；对 PDF 噪声（目录、页码、页眉）做更精细的过滤；考虑引入层级结构（章节树）辅助检索。
  • 检索优化：引入 HyDE（完整版）或 Step-Back Prompting 进行 Query 改写；尝试 ColBERT 等 Late Interaction 模型替代 Bi-Encoder；加入图谱检索或关系型检索增强结构化知识。
  • 精排优化：训练领域特定的 Reranker；引入 LLM-as-a-Reranker（如 GPT-4 打分）对高价值查询精排。
  • 生成优化：引入 Self-RAG（自反思） 机制，让 LLM 判断是否需要检索、检索结果是否相关；加入答案置信度估计。
  • 工程优化：向量库改为 Milvus 支持亿级规模；加入 Redis 缓存高频查询；引入流式推理（Streaming）优化用户体验。
Q10 项目中遇到了哪些挑战？你是如何解决的？进阶
参考回答思路（建议结合自己的实际经历）：

  • 挑战 1 — PDF 解析质量差：汽车手册包含目录、页眉、表格、图片注释，直接解析噪声多。解决方案：多策略解析 + 正则过滤（去目录点、数字页码、极短行），并用三种方案对比互补。
  • 挑战 2 — 检索不准确（召回率低）：用户问题表述与手册文本风格差异大。解决方案：引入 Query 增强（拼接 LLM 生成的摘要答案），同时使用4路召回增加候选多样性。
  • 挑战 3 — LLM 推理速度慢：为对比 5 路召回效果需一题生成 5 个答案，普通推理耗时过长。解决方案：vLLM 批量推理一次性处理 5 个 Prompt，显著降低延迟。
  • 挑战 4 — 显存管理：Embedding 模型 + Reranker + LLM 同时加载显存不足。解决方案：在 Embedding 完成后及时释放（del embeddings + torch.cuda.empty_cache()），分阶段加载。
